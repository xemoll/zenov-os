ZenovOS 0.1.1 — installation and boot guide
=================================================

ZenovOS 0.1.1 uses two images:

- ZenovOS-0.1.1-x86.img   bootable 1.44 MiB FAT12 system image
- ZenovOS-0.1.1-data.img  persistent 16 MiB ATA/ZenovFS data volume

The system boots from the FAT12 image. User files, configuration and native
applications are stored on the separate writable data image and survive a reboot.
This release does not yet install itself to a physical hard disk.

QUICK START
-----------
1. Extract ZenovOS-0.1.1-x86.zip.
2. Keep both IMG files in the same directory.
3. Install QEMU if it is not already available.
4. Run ./run-qemu.sh on Linux/macOS or run-qemu.cmd on Windows.

QEMU — MANUAL COMMAND
---------------------
qemu-system-i386 \
  -drive file=ZenovOS-0.1.1-x86.img,format=raw,if=floppy \
  -drive file=ZenovOS-0.1.1-data.img,format=raw,if=ide,index=0,media=disk \
  -boot a -m 32M

VIRTUALBOX
----------
1. Create a new 32-bit "Other/Unknown" virtual machine.
2. Use legacy BIOS, one CPU and at least 32 MiB RAM.
3. Attach ZenovOS-0.1.1-x86.img to a floppy controller.
4. Attach ZenovOS-0.1.1-data.img as an IDE hard disk.
5. Put the floppy device first in the boot order and start the VM.

These settings match the BIOS/floppy/IDE interfaces used by the system. The automated
release acceptance test runs under QEMU; VirtualBox launch is not part of CI.

MEMORY AND SYSTEM DIAGNOSTICS
-----------------------------
ZenovOS 0.1.1 enables 4 KiB paging, an E820 physical frame allocator and a reusable
split/coalesce kernel heap. Boot self-tests allocate and release multiple physical
frames and fragmented heap blocks before storage and userspace are initialized.

pmm      physical frame counts and managed memory
vm       CR0, CR3 and active kernel/user mappings
mem      E820, heap and free-frame summary
memmap   BIOS E820 regions

PERSISTENT STORAGE
------------------
ZenovFS1 is mounted at /data. File replacement uses a copy-on-write data slot followed
by a sector-atomic metadata commit. Mount recovery discards an uncommitted staging
entry or completes a committed replacement, preserving old-or-new file contents.
Useful commands:

pwd
ls
mkdir PROJECTS
write NOTES.TXT hello
append NOTES.TXT world
cat NOTES.TXT
cp NOTES.TXT COPY.TXT
mv COPY.TXT ARCHIVE.TXT
stat ARCHIVE.TXT
fsck
sync
rm ARCHIVE.TXT
df
mount

NATIVE APPLICATIONS
-------------------
ZenovOS 0.1.1 supports two validated native formats:

- ZEX1, the compact deterministic ZenovOS executable container
- static 32-bit little-endian i386 ELF executables within the user window

Included applications:

run HELLO
run FILEIO.ELF
run ARGS.ELF alpha beta
run CONSOLE.ELF
run PROTECT.ELF
run KACCESS.ELF
run ZENOVAPP.ZEX

FILEIO.ELF validates the file syscall ABI and persistent data. ARGS.ELF validates
argc/argv, stable error codes and guarded userspace pointers. CONSOLE.ELF validates
bounded console input. PROTECT.ELF and KACCESS.ELF deliberately fault to prove that
RX code pages and supervisor state remain protected and that control returns to the
shell. ZENOVAPP.ZEX is compiled from the documented Zenov freestanding subset.

ZEX/ELF support does not provide Windows PE or DOS MZ compatibility. Arbitrary
Windows or DOS .exe files are not supported. Dynamic ELF linking is not supported.

VERIFY THE DOWNLOAD AND PROVENANCE
----------------------------------
Outside the ZIP, SHA256SUMS.txt verifies the boot image, ZIP, build manifest and source
revision file:

sha256sum -c SHA256SUMS.txt

Inside the ZIP:

- IMAGE-SHA256SUMS.txt verifies both images, BUILD-MANIFEST.json and SOURCE-REVISION.txt.
- BUILD-MANIFEST.json records image hashes, the Zenov app ABI, canonical app hash,
  the merged Zenov compiler repository revision and the OS-side compiler source hash.
- SOURCE-REVISION.txt records the exact ZenovOS commit used to create the package.

Run this after extraction:

sha256sum -c IMAGE-SHA256SUMS.txt
